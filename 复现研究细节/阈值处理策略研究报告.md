# 阈值处理策略研究报告

> 生成时间：2026年03月23日
> 作者：Manus AI

## 1. 研究背景

在自动音乐转录（Automatic Music Transcription, AMT）和歌声转录（Singing Voice Transcription, SVT）任务中，基于帧级别的深度学习模型通常会输出一个 $0 \sim 1$ 之间的概率值（如通过 Sigmoid 激活函数），表示在某一帧是否存在特定音高的音符起音（Onset）或持续（Frame）。

为了将这些概率转化为离散的音符事件，必须设定一个或多个阈值（Threshold）进行二值化。阈值的选择直接影响最终评估指标（如 COn、COnP、COnPOff F1 分数）。阈值过低会导致大量误报（False Positives），阈值过高则会导致大量漏报（False Negatives）。

本报告通过查阅 SVT/AMT 领域的主流论文，总结了不同研究团队处理阈值问题的主要策略，并结合当前 `CFH-Transformer` 代码现状给出了改进建议。

## 2. 领域内主流阈值处理策略对比

通过对多篇相关文献的分析，我们将现有的阈值处理策略归纳为三大类：

### 2.1 策略一：固定阈值（通常为 0.5）

这一策略最为简单直接，前提是模型在训练过程中能够充分学习到数据的分布，使得输出概率呈现明显的两极化特征（即有音符的地方接近 1，无音符的地方接近 0）。

- **Onsets & Frames** (Google, ISMIR 2018) [1]：在钢琴 AMT 任务中，直接使用固定阈值 0.5 来决定 Onset 和 Frame 的激活状态。
- **HPPNet** (2022) [2]：在谐波钢琴转录任务中，通过验证集轻量搜索后，固定使用 0.4 作为阈值。
- **VOCANO** (ISMIR 2021) [3]：在 SVT 任务中，输出经过 5-tap 三角滤波器平滑后，直接使用固定阈值 0.5 进行峰值拾取（Peak Picking）。
- **AV-SVT** (NUS, 2023) [4]：在多模态 SVT 任务中，同样使用固定阈值 0.5。其成功应用固定阈值的关键在于训练阶段对 Onset 类别施加了高权重（$w=15.0$），从而缓解了正负样本不平衡的问题。

### 2.2 策略二：验证集网格搜索（Grid Search）

当模型输出概率不够"极端"，或者不同类别的分布差异较大时，研究者倾向于在验证集上搜索最优阈值。

- **High-Resolution Piano Transcription** (Kong et al., 2021) [5]：在验证集上对 Onset 和 Frame 的阈值在 $0.05 \sim 0.60$ 范围内进行网格搜索。
- **Phoneme-Informed SVT** (KAIST, 2023) [6]：在验证集上搜索 $0.1 \sim 0.9$（步长 0.1）的阈值，并结合对标签的三角窗平滑处理。
- **当前代码现状**：在验证集的 10 首歌曲上，穷举搜索 63 种组合（Onset $0.05 \sim 0.50$，Frame $0.05 \sim 0.35$），每 5 个 Epoch 执行一次。

### 2.3 策略三：完全绕开阈值选择

部分研究通过改变任务范式或引入特定的解码算法，从根本上规避了基于帧概率的阈值二值化问题。

- **Pseudo-Label Transfer** (KAIST, ICASSP 2022) [7]：将 SVT 转化为音高估计（Pitch Estimation）问题，直接使用 Softmax + Argmax 获取最高置信度的音高类别，再通过节奏量化转换为离散音符。
- **CTC + Cross-Entropy** (NTU, TASLP 2023) [8]：引入 CTC（Connectionist Temporal Classification）损失。CTC 解码（如 Beam Search）直接输出音符序列，无需精确的帧级别对齐，彻底绕开了帧级别的阈值设定。

## 3. 策略对比汇总表

| 论文/方法 | 任务类型 | 阈值策略 | 是否需要搜索 | 关键辅助技术 |
| :--- | :--- | :--- | :--- | :--- |
| **Onsets & Frames** [1] | 钢琴 AMT | **固定 0.5** | 否 | 联合预测，特征级联 |
| **HPPNet** [2] | 钢琴 AMT | **固定 0.4** | 轻量 | FG-LSTM |
| **High-Res Piano** [5] | 钢琴 AMT | **验证集网格搜索** ($0.05 \sim 0.60$) | 是 | 回归精确的起止时间 |
| **VOCANO** [3] | 歌声 SVT | **固定 0.5** | 否 | 三角滤波平滑 + Peak Picking |
| **Phoneme-Informed** [6] | 歌声 SVT | **验证集搜索** ($0.1 \sim 0.9$) | 是 | 标签平滑 (Label Smoothing) |
| **AV-SVT** [4] | 歌声 SVT | **固定 0.5** | 否 | **Onset 加权 BCE 损失 ($w=15.0$)** |
| **Pseudo-Label** [7] | 歌声 SVT | **完全绕开** | 否 | Pitch Estimation + 节奏量化 |

## 4. 对当前代码的分析与建议

当前代码采用的是**验证集网格搜索**策略，这种做法在逻辑上是严谨的，但也存在明显的缺点：
1. **计算开销大**：每 5 个 Epoch 进行一次搜索，特别是第 1 个 Epoch 时，需要遍历 63 种组合，导致验证过程非常缓慢。
2. **泛化性风险**：由于搜索仅在 10 首验证集歌曲上进行，得出的最优阈值可能过拟合于验证集的分布，在测试集上的表现不够稳健。

### 改进建议

观察当前代码的搜索范围（下探至 0.05），说明模型输出的 Onset 概率整体偏低。这主要是因为在 BCE 损失函数中，Onset、Frame 和 Offset 的权重均为均等（1.0）。然而，在一首歌曲中，Onset 帧的比例极低（通常 $< 5\%$），导致严重的类别不平衡。

**推荐方案：引入加权 BCE 损失 + 减少搜索或固定阈值**

参考 AV-SVT [4] 的做法，建议在损失函数中对 Onset 和 Offset 的正样本赋予更高的权重。

```python
# 建议在 CFTLoss 中的修改示例
onset_loss = F.binary_cross_entropy_with_logits(
    onset_pred, onset_label,
    pos_weight=torch.tensor(5.0).to(device)  # 提高正样本权重，当前为 1.0
)
```

通过这种方式，模型在训练时会更"积极"地预测 Onset，使得输出概率分布更加两极化。一旦概率分布改善：
1. 可以大幅缩小阈值搜索范围（例如只搜索 $0.3 \sim 0.6$）。
2. 甚至可以直接固定阈值为 0.5，从而完全省去耗时的阈值搜索过程，显著提升训练速度。
